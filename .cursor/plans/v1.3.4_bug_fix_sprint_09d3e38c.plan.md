# v1.3.4 Bug Fix Sprint

## Bug 1: ESC Crashes the Game (PauseMenu `_button_tex_pressed`)

**Root cause:** [pause_menu.py](game/ui/pause_menu.py) line 357 references `self._button_tex_pressed`, but it was never defined in `__init`__. The `__init_`_ (line 26-28) defines `_button_tex_normal` and `_button_tex_hover` but not `_button_tex_pressed`. The same pattern in [hud.py](game/ui/hud.py) (line 53) does define all three.

**Fix:** Add `self._button_tex_pressed = "assets/ui/kingdomsim_ui_cc0/buttons/button_pressed.png"` after line 27 in `pause_menu.py`, matching the `hud.py` pattern. The `Button.render` method in [widgets.py](game/ui/widgets.py) (line 227) has a fallback chain (`texture_pressed or texture_hover or texture_normal`), so even if the PNG doesn't exist on disk, the NineSlice renderer falls back to the color-based frame.

**Files:** `game/ui/pause_menu.py` (1-line add in `__init__`)

## Bug 2: Drag-to-Windowed Not Working

**Root cause:** Partial implementation exists from WK7 but was never completed:

- Engine state variables exist (`_borderless_drag_active`, `_borderless_drag_start_pos`, `_borderless_drag_window_offset`) in [engine.py](game/engine.py) lines 81-84
- Drag end (mouseup) and live-drag (mousemotion) handlers exist in [input_handler.py](game/input_handler.py) lines 47-51 and 367-383
- **The mousedown handler that starts the drag was never wired**

**Fix:** Add drag-start logic in `InputHandler.handle_mousedown`:

- When `display_mode` is `"borderless"` or `"fullscreen"` and no menu is open
- If the mousedown is within the top bar region (first ~40px of screen height) and not on the Quit button
- For **borderless**: activate `_borderless_drag_active`, store start position and window offset, then switch to windowed mode
- For **fullscreen**: call `request_display_settings("windowed", window_size)` to drop to windowed mode on the next frame
- Need to check whether `pygame._sdl2.Window` is available (wrapped in try/except) for live-drag positioning

**Files:**

- `game/input_handler.py` (add drag-start in `handle_mousedown`, before world-click handling)
- `game/ui/hud.py` or `game/ui/top_bar.py` (expose a `drag_rect` excluding the Quit button for clean hit-testing)

## Bug 3: Command Bar Buttons Not Clickable

**Root cause:** Two gaps in the click chain:

1. In [command_bar.py](game/ui/command_bar.py) lines 64-77, the **Hire** and **Bounty** button specs have `"action": ""`, so `handle_click` returns `None` for them
2. In [input_handler.py](game/input_handler.py) lines 261-270, the HUD click handler only checks for `"quit"` and `"close_selection"` actions -- it ignores `"build_menu_toggle"`, `"hire_hero"`, and `"place_bounty"` entirely

**Fix:**

Step A -- `game/ui/command_bar.py`: Set proper action strings:

- Hire button (line 68): `"action": "hire_hero"`
- Bounty button (line 75): `"action": "place_bounty"`

Step B -- `game/input_handler.py`: After the existing `"close_selection"` handler (~line 270), add branches for:

- `"build_menu_toggle"` -> `engine.building_list_panel.toggle()` (the panel already has a `toggle()` method)
- `"hire_hero"` -> `engine.try_hire_hero()` (same as K_h handler, line 187)
- `"place_bounty"` -> `engine.place_bounty()` (same as K_b handler, line 210)

**Files:** `game/ui/command_bar.py` (2 string changes), `game/input_handler.py` (add 3 action branches)

## Bug 4: Heroes Over-Prioritizing Inn Over Combat

**Root cause:** Two priority ordering issues in the AI:

1. In [exploration.py](ai/behaviors/exploration.py) `handle_idle` (~~line 238), the **get-a-drink** check (12% RNG roll) runs **before** the **enemies-nearby** check (~~line 283). When the roll succeeds, the hero heads to the Inn even with monsters 5 tiles away.
2. In [basic_ai.py](ai/basic_ai.py), `send_home_to_rest` runs **before** the state machine reaches `handle_idle`, with no nearby-enemy guard. A hero that took enough damage will route to the Inn (if closer than guild) without considering enemies in the area.

**Fix:**

Step A -- `ai/behaviors/exploration.py`: Gate the get-a-drink block with a nearby-enemy check. Either:

- Move the enemies-nearby engagement block above get-a-drink, OR
- Add `and not enemies_nearby` to the get-a-drink condition (simpler, lower risk)

Step B -- `ai/basic_ai.py`: Before calling `send_home_to_rest`, check for enemies within a combat radius (e.g., 5 tiles / 160px). If enemies are nearby, skip rest and let the hero engage. Exception: if the hero is critically low HP (e.g., below 25%), allow rest even with enemies nearby (survival instinct).

**Files:** `ai/behaviors/exploration.py` (reorder or guard), `ai/basic_ai.py` (add nearby-enemy check before rest routing)
