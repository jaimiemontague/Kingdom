# v1.6 Round 4 — Interpolation Fix & Smoothness Monitor

**Date:** 2026-05-17  
**Prerequisite:** v1.6.4 (Rounds 1-3 shipped)  
**Symptom:** Gameplay doesn't feel smoother despite FPS improvements; FPS counter shows decent numbers but human perception disagrees  
**Root Causes:**  
1. R2-D's sync_key early-out defeats R1's lerp interpolation — entities lurch at 20 Hz then freeze  
2. Lerp uses constant alpha (0.82) creating exponential decay, not uniform motion  
3. FPS counter shows EMA average, hiding the frame-time variance that humans perceive  

---

## DIAGNOSIS

### Bug: sync_key Defeats Position Interpolation

In `ursina_renderer.py:422-425`, the R2-D "skip unchanged" optimization returns early before the lerp code at line 435 can run:

```python
sync_key = (pos_xyz, clip_name, frame_idx, scale_xyz)
if getattr(ent, '_ks_last_sync_key', None) == sync_key:
    return  # ← EXITS before position lerp at line 435!
```

Between 20 Hz sim ticks, `pos_xyz` doesn't change. If the animation frame also hasn't changed (typical for 6-12 FPS pixel art), the sync_key matches and the function returns without running the lerp. Result:

- Sim tick frame: lerp runs, entity moves 82% toward target
- Next 2 render frames: sync_key matches → early return → entity frozen
- Next sim tick: lerp runs again → visible jump

Entities move in a 20 Hz "lurch then freeze" pattern even at 60 FPS render.

### Bug: Exponential Lerp Creates Non-Uniform Motion

Even without the sync_key bug, `alpha = 0.82` as a constant blend factor creates exponentially decaying movement:

- Frame 1 after tick: moves 82.0% of distance
- Frame 2: moves 14.8% of distance (5.5x less)
- Frame 3: moves 2.7% of distance (30x less)

This "lurch then crawl" is perceivable. True interpolation should produce uniform motion between sim ticks.

### Problem: FPS Counter Measures Wrong Metric

`pygame.time.Clock.get_fps()` (render_coordinator.py:232) is an EMA over ~1 second. The Ursina path uses `prev * 0.92 + inst * 0.08` (ursina_app.py:1165). Both show average frame rate, which hides frame-time variance — the thing humans actually perceive as stutter.

A game at 55 FPS with periodic 100ms hitches feels terrible but shows "55 FPS." A steady 30 FPS feels smooth but shows "30 FPS." The counter cannot distinguish these.

## FIX 1: Linear Interpolation with Sim Blend Fraction

**Goal:** Replace the broken exponential lerp with proper linear interpolation using the sim accumulator as blend factor, producing perfectly uniform visual motion between 20 Hz sim ticks.

### Approach: Render-One-Tick-Behind Interpolation

Standard game-industry technique. The renderer interpolates between the **previous** sim tick position and the **current** sim tick position using `blend = sim_accumulator / FIXED_SIM_DT`:

```
visual_pos = prev_sim_pos + (curr_sim_pos - prev_sim_pos) * blend
```

- At `blend = 0.0` (just after a sim tick): visual = prev_sim_pos
- At `blend = 1.0` (about to tick again): visual = curr_sim_pos
- Between: linear progression → perfectly uniform motion

This adds one sim tick of visual latency (50ms at 20 Hz) — imperceptible in a top-down strategy game.

### Step 1A: Add `sim_blend_fraction` to SimStateSnapshot

**File:** `game/sim/snapshot.py`

Add field to the frozen dataclass:

```python
sim_blend_fraction: float = 0.0
```

### Step 1B: Compute blend fraction in `build_snapshot()`

**File:** `game/engine.py` — `build_snapshot()` method (line 1678)

Before calling `self.sim.build_snapshot(...)`, compute:

```python
blend = self._sim_accumulator / self._FIXED_SIM_DT if self._FIXED_SIM_DT > 0 else 0.0
blend = max(0.0, min(1.0, blend))
```

Pass `sim_blend_fraction=blend` through `self.sim.build_snapshot(...)`.

**File:** `game/sim_engine.py` — `build_snapshot()` method (line 378)

Add `sim_blend_fraction: float` parameter, pass it through to `SimStateSnapshot(...)`.

### Step 1C: Pass blend fraction through renderer update chain

**File:** `game/graphics/ursina_renderer.py`

In `update(self, snapshot)` (line 538): extract `self._frame_blend = getattr(snapshot, 'sim_blend_fraction', 0.0)` at the top.

Pass `self._frame_blend` to all `_sync_unit_atlas_billboard()` calls (heroes, enemies, peasants, guards, tax collector).

### Step 1D: Restructure `_sync_unit_atlas_billboard` — separate position from appearance

**File:** `game/graphics/ursina_renderer.py` — method at line 403

Restructure to two phases:

**Phase 1 — Position interpolation (always runs, every frame):**

```python
# Track previous/current sim positions for interpolation
curr_sim = getattr(ent, "_ks_curr_sim_pos", None)
if curr_sim is None or curr_sim != pos_xyz:
    # Sim position changed → new sim tick arrived (or first frame)
    ent._ks_prev_sim_pos = curr_sim if curr_sim is not None else pos_xyz
    ent._ks_curr_sim_pos = pos_xyz

prev_sim = ent._ks_prev_sim_pos
curr_sim = ent._ks_curr_sim_pos

# Linear interpolation between prev and current sim positions
dx = curr_sim[0] - prev_sim[0]
dy = curr_sim[1] - prev_sim[1]
dz = curr_sim[2] - prev_sim[2]
dist_sq = dx * dx + dy * dy + dz * dz

if dist_sq < 0.0001 or dist_sq > 9.0:
    # Too close (already there) or too far (teleport) → snap
    interp_pos = curr_sim
else:
    t = blend_fraction
    interp_pos = (
        prev_sim[0] + dx * t,
        prev_sim[1] + dy * t,
        prev_sim[2] + dz * t,
    )

if getattr(ent, "_ks_last_pos", None) != interp_pos:
    ent.position = interp_pos
    ent._ks_last_pos = interp_pos
```

**Phase 2 — Appearance updates (skippable via sync_key, minus pos_xyz):**

```python
clip_name, frame_idx = self._compute_anim_frame(...)

# sync_key no longer includes pos_xyz — position is handled above
appearance_key = (clip_name, frame_idx, scale_xyz)
if getattr(ent, '_ks_last_appearance_key', None) == appearance_key:
    return
ent._ks_last_appearance_key = appearance_key

# ... UV, scale, color, shader updates (unchanged) ...
```

This preserves the R2-D optimization for texture/scale/color work while ensuring position interpolation runs every frame.

### Step 1E: Remove old lerp attributes

Remove references to `_ks_visual_pos` and the old `_ks_last_sync_key`. Replace with `_ks_prev_sim_pos`, `_ks_curr_sim_pos`, `_ks_last_appearance_key`.

### Edge Cases

- **First frame (no previous position):** `curr_sim is None` → snap to pos_xyz
- **Teleport / enter building (distance > 3.0 world units):** `dist_sq > 9.0` → snap
- **Entity stationary:** `dist_sq < 0.0001` → snap (no interpolation of zero motion)
- **Paused game:** accumulator doesn't advance → blend stays constant → no visual movement (correct)
- **Speed multiplier changes:** accumulator scales by speed multiplier → interpolation naturally adapts
- **Camera frustum re-entry:** entity was `enabled=False`, comes back on-screen → `_ks_curr_sim_pos` may be stale. When pos_xyz differs from stored curr_sim, both prev and curr update → entity snaps to current position (correct for re-entry)

## FIX 2: Smoothness Monitor (replaces basic FPS overlay)

**Goal:** Replace the average-FPS overlay with a frame-time-focused monitor that measures what humans perceive. Toggled with the same F2 key.

### Step 2A: Frame Time Ring Buffer — data collection

**File:** `game/engine_facades/render_coordinator.py`

Add a ring buffer to collect per-frame timing data. Initialize in `__init__`:

```python
self._frame_times = []       # Ring buffer of last 300 frame times (ms)
self._frame_times_max = 300  # ~5 seconds at 60 FPS
self._frame_sim_flags = []   # Parallel buffer: True if a sim tick ran this frame
self._smoothness_panel = None
self._smoothness_dirty = True
```

Each frame (at the top of `render_perf_overlay`), record the frame time:

```python
frame_ms = e.clock.get_rawtime()  # Raw ms for LAST frame (no EMA)
self._frame_times.append(frame_ms)
if len(self._frame_times) > self._frame_times_max:
    self._frame_times.pop(0)
```

Note: `pygame.time.Clock.get_rawtime()` returns the actual milliseconds of the previous `clock.tick()` call, WITHOUT smoothing. This is the true frame time.

### Step 2B: Compute Smoothness Metrics

Compute these from the ring buffer every 250ms (matching existing update interval):

1. **P50 (median):** Middle value when frame times are sorted
2. **P95 (1-in-20):** 95th percentile — noticeable hitches
3. **P99 (1-in-100):** 99th percentile — worst-case freezes
4. **1% Low FPS:** Average FPS of the worst 1% of frames: `1000.0 / mean(worst_1pct_frame_times)`
5. **Jitter:** Standard deviation of frame times over the window
6. **Sim tick ratio:** Fraction of frames that ran a sim tick (should be ~33% at 60 FPS / 20 Hz sim)

Implementation:

```python
import statistics

sorted_times = sorted(self._frame_times)
n = len(sorted_times)
if n >= 10:
    p50 = sorted_times[n // 2]
    p95 = sorted_times[int(n * 0.95)]
    p99 = sorted_times[int(n * 0.99)]
    worst_1pct = sorted_times[int(n * 0.99):]
    low_1pct_fps = 1000.0 / (sum(worst_1pct) / len(worst_1pct)) if worst_1pct else 0
    jitter = statistics.stdev(self._frame_times) if n >= 2 else 0
```

### Fix 1b: Tick-Counter Window Advancement (Post-Playtest Hotfix)

**Bug found in playtest:** The interpolation window (prev/curr sim positions) only advanced when `pos_xyz` changed. Once an entity stopped moving (e.g., hero enters inn, peasant stands still), prev was stuck at the old position while curr held the new one. The blend fraction cycling (0.2 → 0.9 → 0.2) caused the visual position to oscillate between prev and curr — producing 100-200px vertical jitter for heroes in buildings and rapid twitching for peasants/tax collector.

**Fix:** Added `sim_tick_id` (monotonic counter, incremented per sim tick) to `SimStateSnapshot`. The renderer now advances the interpolation window on every sim tick boundary regardless of whether position changed:

```python
if last_tick != self._frame_tick_id:
    ent._ks_prev_sim_pos = ent._ks_curr_sim_pos   # prev advances to old curr
    ent._ks_curr_sim_pos = pos_xyz                  # curr gets new (or same) position
    ent._ks_last_tick_id = self._frame_tick_id
```

When an entity is stationary: prev==curr → interpolation = curr regardless of blend → no jitter.
