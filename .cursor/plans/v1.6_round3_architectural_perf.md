# v1.6 Round 3 — Architectural Performance Optimizations

**Date:** 2026-05-17  
**Prerequisite:** v1.6.3 (Round 2 shipped — atlas, pathfinding budget, throttles)  
**Symptom:** 10-20 FPS at normal/fast speeds during POI exploration  
**Root Cause:** Panda3D scene graph traversal of 16k-44k nodes + no spatial culling + sim scales linearly with game speed

---

## DIAGNOSIS

### Why Still 10-20 FPS After Rounds 1-2?

Round 1-2 eliminated per-frame texture uploads (8-12ms) and pathfinding storms (unbounded spikes). The remaining cost breakdown:

```
Panda3D scene graph traversal (16k-44k nodes)   10-20 ms  ← DOMINANT
Python entity sync loops (all entities/frame)     3-8 ms
Sim update at 2x speed (all entities × 2)         4-8 ms
Entity separation (170 entities, every 2 frames)  1-3 ms
Fog/tree throttled updates                         0-2 ms
─────────────────────────────────────────────────────────
TOTAL                                            18-41 ms = 24-55 FPS theoretical
                                                          = 10-20 FPS observed (spikes)
```

The scene graph has:
- ~6,250-12,500 tree entities (one per TREE tile on 250×250 map)
- ~3,125-12,500 grass scatter entities (doodads, flowers, clumps)
- ~3,125-9,375 path/water tile entities
- ~50-500 building entities
- ~100-150 unit billboards (heroes, enemies, peasants, guards)

Panda3D traverses ALL of these for transform updates and draw-call submission every frame, even if:
- They're static and haven't moved
- They're behind the camera (no frustum culling from Python side)
- They're far away and sub-pixel

---

## FIXES (Priority Order)

### Fix 1: Camera Frustum Culling (Python-Side)

**Severity:** CRITICAL — eliminates 70-90% of per-frame sync work  
**Risk:** LOW — pure optimization, no behavioral changes

**Problem:** The renderer syncs ALL entities every frame regardless of camera position. At any given time, only ~10-30% of the 250×250 map is visible.

**Solution:**
1. Compute visible tile bounding box from camera position each frame
2. In all entity sync loops (buildings, units, trees), skip entities whose tile position is outside the visible rect + a small margin
3. For units entering/leaving view: show/hide their Ursina Entity (O(1) Panda3D call)

**Implementation:**
- Add `get_visible_tile_rect()` method to the renderer that returns `(min_tx, min_ty, max_tx, max_ty)` based on camera world position and zoom level
- Add margin of 3-5 tiles beyond visible edges (prevents pop-in during camera pan)
- In `_sync_snapshot_heroes()`, `_sync_snapshot_enemies()`, `_sync_snapshot_peasants()`, `_sync_snapshot_guards()`: early-continue if entity tile is outside visible rect
- In `_sync_snapshot_buildings()`: early-continue for out-of-view buildings
- Entities going off-screen: set `ent.enabled = False` (hides from Panda3D render, zero traversal cost)
- Entities coming on-screen: set `ent.enabled = True` and sync position/UV

**Expected gain:** Python sync loops drop from ~100-500 iterations to ~20-80. Scene graph active nodes drop from 16k+ to ~2k-4k visible.

---

### Fix 2: Static Terrain Flattening

**Severity:** HIGH — reduces Panda3D draw calls from thousands to tens  
**Risk:** MEDIUM — must preserve fog-of-war visibility gating

**Problem:** Each grass clump, path tile, rock, water tile is a separate Ursina Entity with its own transform node in the scene graph. Panda3D must traverse all of them even if they're static.

**Solution:**
- Group terrain entities by **tile visibility state** (UNSEEN, SEEN, VISIBLE)
- Call `flatten_strong()` on groups of same-model static entities that share a visibility state
- For fog-of-war transitions: instead of per-entity show/hide, swap entire batch visibility

**Alternative (simpler):** Since Fix 1 (frustum culling) already hides off-screen terrain by disabling entities, Fix 2 may be OPTIONAL if Fix 1 provides sufficient FPS gain. Implement Fix 2 only if Fix 1 alone doesn't reach target FPS.

**Implementation (if needed):**
- After `build_3d_terrain()` completes, group all grass scatter entities by tile into chunks (e.g., 16×16 tile chunks)
- Each chunk is a parent Entity; call `chunk.combine()` or `chunk.flatten_strong()` (Panda3D merges all children into one node)
- Fog updates toggle chunk visibility instead of individual entities
- Trees remain individual (they have growth animation / scale changes)

**Expected gain:** 16k static entities → ~250 chunk nodes. Panda3D traversal drops from 10-20ms to 2-4ms.

---

### Fix 3: Fixed-Rate Simulation Tick

**Severity:** MEDIUM — bounds sim cost per frame, prevents speed-scaling overhead  
**Risk:** MEDIUM — changes core update loop, needs careful testing

**Problem:** At 2x-3x game speed, the sim processes 2-3x more entity updates per real-time frame. Pathfinding replans happen 2-3x more often. Combat/AI checks scale linearly with speed.

**Solution:**
- Run simulation at a **fixed tick rate** (e.g., 20 Hz = 50ms per tick, regardless of game speed)
- Game speed controls **how many sim ticks run per real frame** (not how big dt is)
- At 1x speed: 1 tick/frame. At 2x speed: 2 ticks/frame. At 3x speed: 3 ticks/frame.
- Each tick uses a FIXED dt (50ms), so per-tick cost is constant
- Cap at max 4 ticks/frame to prevent death spiral at low FPS
- Renderer already has position interpolation (from Round 1) — it smooths between ticks automatically

**Implementation:**
- In `GameEngine.tick_simulation()`: accumulate real dt into a sim accumulator
- While accumulator >= FIXED_TICK_MS: run one sim tick with FIXED_TICK_MS, decrement accumulator
- Cap loop at MAX_TICKS_PER_FRAME (e.g., 4)
- Render interpolation already reads entity positions and lerps — no change needed there
- Pathfinding budget resets per SIM TICK (not per render frame) — budget per tick may need adjustment

**Expected gain:** Bounded per-frame sim cost. At 3x speed, worst case is 3 × (fixed tick cost) instead of 1 × (3x scaled dt cost). Eliminates speed-dependent behavioral bugs.
