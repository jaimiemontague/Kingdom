# v1.6 Performance Analysis

**Date:** 2026-05-16  
**Symptoms:** 10-20 FPS, heroes "spring forward" every few steps, CPU >> GPU usage but only ~20% CPU  
**Environment:** Ursina/Panda3D engine, Python, Windows 10, dedicated NVIDIA GPU confirmed active

---

## Why 20% CPU but Low FPS?

The game is **single-threaded** (Python GIL). On a modern 4-8 core CPU, saturating one core = 12-25% total CPU usage in Task Manager. The single render/sim thread is at or near 100% utilization — the other cores sit idle. GPU is starved because the CPU can't feed it draw calls fast enough.

---

## TOP 3 PERFORMANCE ISSUES

### Issue #1: Per-Frame Texture Pipeline (CPU-Bound PIL/Pygame to GPU)

**Severity:** CRITICAL — estimated 8-12ms per frame (50-70% of frame budget)

**Location:** `game/graphics/ursina_renderer.py` lines 610-626 (heroes), 656-659 (enemies), 695-698 (peasants), 728-731 (guards)

**What's happening:**  
Every frame, for every visible unit (~66 entities typical: 10 heroes + 20 enemies + 30 peasants + 5 guards + 1 tax collector), the renderer:

1. Calls `_unit_anim_surface()` which uses `time.time()` (low-precision on Windows, ~15ms granularity) to compute animation frame index
2. Returns a new pygame Surface from `clip.frames[idx]`
3. Calls `TerrainTextureBridge.surface_to_texture()` which:
   - Runs `pygame.image.tobytes()` (CPU-bound byte conversion)
   - Creates a `PIL.Image.frombytes()` object (CPU-bound)
   - Uploads the texture to GPU (synchronous stall)

**Impact:** ~66 pygame→PIL→GPU texture conversions per frame = **~1,980 texture uploads/second**. All on the main thread. PIL operations are pure Python with no parallelism.

**Additional sub-issues:**
- `clips_for()` called per-unit per-frame (lines 600, 647, 686) — rebuilds/copies clip dictionaries unnecessarily
- `sync_billboard_entity()` unconditionally sets `ent.position = pos_xyz` every frame, even for stationary units, triggering Panda3D transform invalidation
- `time.time()` has 15ms granularity on Windows causing animation frame jitter

**Why instancing is disabled:** `KINGDOM_URSINA_INSTANCING=1` is opt-in due to visual bugs. Even enabled, still loops all units to build instance buffers.

### Issue #2: Pathfinding Storms with No Global Frame Budget

**Severity:** CRITICAL — estimated 5-15ms spike every 250ms, primary cause of "spring forward"

**Locations:**
- `game/entities/hero.py` lines 819-842 (hero replan logic)
- `game/systems/pathfinding.py` lines 84-203 (A* implementation)
- `game/sim/hero_guardrails_tunables.py` line 46 (`PATH_REPLAN_MIN_INTERVAL_MS = 250`)

**What's happening:**  
Each hero independently decides when to replan its path. The rate limit (`PATH_REPLAN_MIN_INTERVAL_MS = 250ms`) is **per-hero, not global**. With 30 heroes, multiple heroes hit their replan window in the same frame, triggering simultaneous A* searches.

- A* on a 250x250 grid with max 8,000 node expansions = 2-10ms per search
- 5-15 heroes replanning in the same frame = **10-50ms frame stall**
- During the stall: no movement updates, no rendering
- Next frame: all pending movement catches up at once → **visible "spring forward"**

**Contributing factors:**
- Enemy pathfinding uses floating-point world coords as goal keys (`enemy.py` line 254: `goal_key = (int(target_x), int(target_y))`) instead of grid-snapped coords, causing unnecessary replans from sub-tile position changes
- Path cache validates every cached cell's walkability on access (pathfinding.py lines 135-145), adding overhead even for cache hits
- No preemption or time-slicing — once A* starts, it runs to completion

### Issue #3: Frame-Rate Dependent Movement + No Render Interpolation

**Severity:** HIGH — directly causes the visual "spring forward" artifact

**Locations:**
- `game/entities/hero.py` line 736: `move_dist = self.speed * dt * 60`
- `game/systems/navigation.py` lines 92-102 (step_towards)
- `game/graphics/ursina_renderer.py` (no interpolation between sim positions)

**What's happening:**  
Movement is calculated as `speed * dt * 60`, which assumes 60 FPS baseline. At 10-15 FPS:
- `dt` = 0.066-0.1s instead of expected 0.016s
- Movement per frame is 4-6x larger than at 60 FPS
- Combined with pathfinding stalls (Issue #2), this creates a pattern of:
  - **Frame N:** Pathfinding stall, no movement (hero frozen)
  - **Frame N+1:** Large `dt` catchup, hero moves 4-6x normal distance (spring forward)

There is **no render-side interpolation** between simulation positions. The renderer reads `entity.x, entity.y` directly each frame. When the simulation hitches and then catches up, the entity visually teleports.

**The full "spring forward" sequence:**
1. Hero needs new path → A* blocks frame for 5-10ms
2. Frame takes 50-100ms total (10-20 FPS)
3. Next frame: `dt` is large, `move_dist = speed * 0.066 * 60` = large jump
4. Position updates directly, no smoothing → visual pop
5. Animation uses `time.time()` independently → animation frame can also skip, compounding the jitter

## SECONDARY ISSUES

### Entity Separation O(N) Every Frame
**Location:** `game/sim_engine.py` lines 790-851  
- Spatial grid with 16px cells, 3x3 neighborhood checks
- 40-100+ entities × `math.sqrt()` per pair = 2-3ms/frame
- Called unconditionally every frame regardless of entity movement

### All Entities Updated Every Frame Without Culling
**Location:** `game/sim_engine.py` lines 548-705  
- 30 heroes + 100 enemies + 5 peasants + 40 buildings + 1000+ trees = ~1,175 entities
- No distance-based update throttling
- No frustum culling for simulation updates
- Dead-entity cleanup creates new lists every frame: `self.enemies = [e for e in self.enemies if e.is_alive]`

### POI Interaction O(Heroes x POIs) Every Frame
**Location:** `game/systems/poi_interaction.py` lines 68-125  
- 30 heroes × 15 POIs = 450 distance checks per frame
- Knowledge cascade search (lines 264-276) iterates ALL undiscovered POIs per interaction

### Fog Overlay Texture Rebuild
**Location:** `game/graphics/ursina_terrain_fog_collab.py` lines 50-240  
- Every fog revision: allocates 90KB bytearray, creates pygame Surface, converts to PIL, uploads to GPU
- `sync_visibility_gated_terrain()` loops ALL prop entities (potentially 5000+) on fog revision change

## ROOT CAUSE SUMMARY

```
Frame Budget @ 60 FPS = 16.67ms
Frame Budget @ 30 FPS = 33.3ms

Actual per-frame costs (estimated):
  Texture pipeline (66 units × PIL)     8-12 ms  ← DOMINANT
  Pathfinding (avg 5 replans/frame)     5-10 ms  ← SPIKY
  Entity updates (1175 entities)         3-5 ms
  Entity separation                      2-3 ms
  Panda3D render submit                  5-8 ms
  HUD + fog texture upload               2-3 ms
  ─────────────────────────────────────────────
  TOTAL                                25-41 ms  = 24-40 FPS theoretical
                                                 = 10-20 FPS with spikes
```

The 10-20 FPS observed (vs. 24-40 theoretical) is explained by pathfinding **storms** — frames where multiple heroes replan simultaneously push individual frames to 60-100ms.

## RECOMMENDED FIXES (Priority Order)

### P0 — Texture Atlas for Unit Animation
Replace per-unit-per-frame PIL conversion with a pre-baked texture atlas. All animation frames pre-rendered to a single GPU texture at startup. Per-frame cost drops from ~66 PIL conversions to ~66 UV coordinate updates (essentially free).

**Expected gain:** 8-12ms/frame → <1ms/frame = +20-30 FPS

### P1 — Global Pathfinding Frame Budget
Add a per-frame pathfinding queue with time budget (e.g., max 3ms/frame for all pathfinding). Heroes waiting for paths continue following their last path or use direct steering. Stagger replans across frames.

**Expected gain:** Eliminates spikes, smooths frame times, removes "spring forward"

### P2 — Render-Side Position Interpolation
Store `prev_position` and `curr_position` on each entity. Renderer interpolates between them based on frame fraction. Decouples visual smoothness from simulation tick rate.

**Expected gain:** Eliminates visual "spring forward" even if sim runs at lower tick rate

### Bonus — Quick Wins
- Cache `clips_for()` at unit spawn (save 2-4ms/frame)
- Skip `ent.position =` when unchanged (save 1-2ms/frame)
- Fix `move_dist = speed * dt` without the `* 60` (frame-rate independence)
- Increase `PATH_REPLAN_MIN_INTERVAL_MS` from 250 to 500-1000ms as interim bandaid
- Enable instancing (`KINGDOM_URSINA_INSTANCING=1`) and fix the visual bugs

## FIXES APPLIED (2026-05-16)

### Fix 1: Texture Atlas Pipeline
**Files:** `game/graphics/unit_atlas.py`, `game/graphics/ursina_renderer.py`, `game/graphics/ursina_entity_render_collab.py`

- Single 2048x2048 atlas texture shared across all unit billboards (543 frames packed)
- Per-frame cost: UV offset update (vec2 uniform) instead of texture rebind
- `_compute_anim_frame()` uses `time.perf_counter()` (not `time.time()`)
- `_get_cached_clips()` caches clip dicts at renderer level
- Position/scale/color writes guarded with change detection

### Fix 2: Pathfinding Frame Budget
**Files:** `game/systems/navigation.py`, `game/sim_engine.py`, `game/entities/hero.py`, `game/sim/hero_guardrails_tunables.py`

- Global 3ms/frame budget for all A* calls (`PathfindingBudget` class)
- `begin_frame()` called at top of sim update, resets budget
- Returns empty path (hero keeps existing path) when budget exhausted
- Per-hero stagger: 0-200ms offset via `crc32(hero_id)`, spreads replans across frames
- `PATH_REPLAN_MIN_INTERVAL_MS` raised from 250ms to 400ms (effective 400-600ms with stagger)

### Fix 3: Render Position Interpolation
**File:** `game/graphics/ursina_renderer.py` (`_sync_unit_atlas_billboard`)

- Lerp smoothing (alpha=0.82) between previous and target visual position
- Snap threshold: <0.01 units (already there) or >3.0 units (teleport/enter building)
- Eliminates visual jumps from frame drops without adding perceivable latency

### Post-Fix Metrics (headless stress test, 30 heroes, 600 ticks)
```

P50:       0.48ms
P95:       2.15ms  (was: unbounded spikes from pathfinding storms)
P99:       3.17ms  (bounded by 3ms pathfinding budget)
Max:       10.85ms (initial fog/nature, not pathfinding)
Headroom:  15.62ms within 60fps budget (sim only)
```

**Expected live FPS improvement:** 10-20 FPS → 40-60+ FPS (pending visual verification)
