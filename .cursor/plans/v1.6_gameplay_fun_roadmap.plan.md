# v1.6.0 Roadmap — "Make It Fun" (Gameplay Challenge & Polish)

**Created:** 2026-05-19 | **PM:** Agent 01 | **Current version:** Prototype v1.5.8
**Goal:** Transform Kingdom Sim from a tech demo into an actually fun, challenging game with real tension, resource pressure, and satisfying combat loops. v1.6.0 is the first version a player could sit down with and have a good time.

---

## Vision

Right now the game is a beautiful sandbox with no teeth. Monsters trickle in, heroes are cheap and unlimited, there's no resource pressure, and the player never feels threatened. v1.6 changes that by:

1. **Making monsters dangerous** — wave events that create "oh no" moments, lairs that are real threats
2. **Making heroes scarce and valuable** — capped per guild, expensive, so you care when one dies
3. **Making gold matter** — doubled hero costs + starting buildings eat into starting gold, forcing real economic decisions
4. **Making defense matter** — guardhouse shoots arrows so it's worth building near your economy
5. **Making bounty flags visible** — so the core player-to-hero communication tool actually works in 3D
6. **Giving the player a real start** — pre-placed guilds + market + guardhouse so gameplay begins immediately
7. **Letting players own the challenge** — Easy/Normal/Hard difficulty with mid-game switching + lock option
8. **Making dev testing painless** — dev mode with free heroes so we can playtest new features fast

---

## Architecture: Two-Phase Rollout

### Phase 1: Sprint to v1.5.9 (Implementation Sprint)
Ship all mechanical changes. Focus on getting features working, not perfect balance.

### Phase 2: Playtest to v1.6.0 (Tuning Sprint)
Jaimie playtests v1.5.9 extensively. We iterate on spawn rates, gold costs, wave timing, and difficulty curve until it *feels* right. Once it's fun, we stamp it v1.6.0.

### Phase 3: v1.6.x (Post-Launch Depth)
Hero development features from the existing roadmap: items & equipment, LLM personality flush-out, named bosses, quest hooks, hero relationships. These add depth to the fun foundation v1.6.0 establishes.

### Feature 1: Wave Events System (Option B from Analysis)
**Owner:** Agent 05 (GameplaySystemsDesigner) primary, Agent 03 (TechnicalDirector) support
**Intelligence:** Agent 05 HIGH, Agent 03 MEDIUM

**What:** Layer a discrete "wave event" system on top of the existing trickle spawner. Every N minutes, a named event fires with a burst of themed enemies.

**Design:**

New file: `game/systems/wave_events.py` (or extend `spawner.py`)

```
WaveEventSystem(GameSystem):
    - Fires named events on a timer (first event at ~3 minutes, then every 2-4 minutes)
    - Each event has:
        - name: str ("Goblin Raid", "Skeleton Legion", "Spider Swarm", "Wolf Pack", "Bandit Ambush")
        - enemy_composition: list of (enemy_type, count)
        - spawn_direction: "random_edge" | "nearest_lair" | "all_edges"
        - warning_seconds: 10 (HUD toast + horn SFX before arrival)
        - reward_gold: int (bonus gold for surviving/clearing the wave)
    - Escalation: events get harder over time (bigger bursts, mixed compositions, eventually boss waves)
    - Separate cap from lair spawns: wave events can temporarily exceed MAX_ALIVE_ENEMIES by up to 50%
```

### Feature 8: Difficulty Setting (Easy / Normal / Hard)
**Owner:** Agent 05 (GameplaySystemsDesigner) primary, Agent 08 (UX_UI_Director) for menu UI
**Intelligence:** Agent 05 MEDIUM, Agent 08 MEDIUM

**What:** A runtime difficulty system that scales monster-related values via multipliers. The player can change difficulty mid-game from a menu — it only affects things that adjust cleanly at runtime (spawn rates, enemy stats, wave sizes). It does NOT touch gold costs, hero costs, or building prices. There's also a "Lock Difficulty" button for players who want commitment.

**Design:**

New file: `game/systems/difficulty.py`

```
class DifficultyLevel(Enum):
    EASY = "easy"
    NORMAL = "normal"
    HARD = "hard"

class DifficultySystem:
    current: DifficultyLevel = DifficultyLevel.NORMAL
    locked: bool = False

    # Multiplier tables — applied at runtime to base values
    MULTIPLIERS = {
        DifficultyLevel.EASY: {
            "spawn_interval": 1.5,      # 50% slower spawns
            "enemies_per_wave": 0.6,     # 40% fewer enemies per wave
            "enemy_hp": 0.7,             # 30% less HP
            "enemy_damage": 0.7,         # 30% less damage
            "wave_event_count": 0.6,     # 40% fewer enemies in wave events
        },
        DifficultyLevel.NORMAL: {
            "spawn_interval": 1.0,
            "enemies_per_wave": 1.0,
            "enemy_hp": 1.0,
            "enemy_damage": 1.0,
            "wave_event_count": 1.0,
        },
        DifficultyLevel.HARD: {
            "spawn_interval": 0.7,       # 30% faster spawns
            "enemies_per_wave": 1.5,     # 50% more enemies per wave
            "enemy_hp": 1.3,             # 30% more HP
            "enemy_damage": 1.3,         # 30% more damage
            "wave_event_count": 1.5,     # 50% more enemies in wave events
        },
    }

    def get_multiplier(self, key: str) -> float: ...
    def set_difficulty(self, level: DifficultyLevel) -> bool:
        if self.locked: return False
        ...
    def lock_difficulty(self) -> None: ...
```

**What scales with difficulty (monsters only):**
- Lair spawn intervals (multiplied — higher = slower)
- Number of enemies per lair spawn tick (multiplied, rounded)
- Wave event enemy counts (multiplied, rounded)
- Trickle spawner enemies per wave (multiplied, rounded)
- Enemy HP on spawn (base HP × multiplier)
- Enemy attack damage on spawn (base attack × multiplier)

**What does NOT scale (economy stays fixed):**
- Hero hire cost
- Building costs
- Starting gold
- Guild hero cap
- Bounty costs/rewards
- Gold drops from enemies
- Tax rates

**Integration points:**
- `spawner.py`: multiply `spawn_interval` and `enemies_per_wave` by difficulty multipliers
- `lair.py`: multiply spawn interval and pack size by difficulty multipliers
- `wave_events.py`: multiply enemy counts in wave compositions by difficulty multiplier
- `enemy.py`: on enemy creation, apply HP and damage multipliers from current difficulty
- All systems read multipliers from a shared `DifficultySystem` instance (passed via `SystemContext` or similar)

**Menu UI (Agent 08):**
- Difficulty selector in the ESC/pause menu (not the main HUD — this is a settings-level control)
- Three buttons: Easy / Normal / Hard, with the current selection highlighted
- Current difficulty shown as a small label somewhere visible (e.g., top-right corner or pause menu header)
- "Lock Difficulty" button below the selector
- When "Lock Difficulty" is clicked, show a confirmation dialog:
  > **Lock Difficulty?**
  > "Once you lock difficulty, it can't be unlocked for the remaining duration of the game. Are you sure?"
  > [Yes, Lock It] [Cancel]
- After locking: difficulty buttons become grayed out/disabled, lock button changes to "Difficulty Locked" (disabled)
- Difficulty changes take effect immediately on the next spawn tick / wave event (no need to restart)

**Config additions:**
```python
@dataclass(frozen=True)
class DifficultyConfig:
    default_difficulty: str = "normal"  # easy, normal, hard
    easy_spawn_interval_mult: float = 1.5
    easy_enemy_count_mult: float = 0.6
    easy_enemy_hp_mult: float = 0.7
    easy_enemy_damage_mult: float = 0.7
    hard_spawn_interval_mult: float = 0.7
    hard_enemy_count_mult: float = 1.5
    hard_enemy_hp_mult: float = 1.3
    hard_enemy_damage_mult: float = 1.3
```

**Acceptance:**
- [ ] Difficulty defaults to Normal
- [ ] Changing to Easy makes spawns slower and enemies weaker
- [ ] Changing to Hard makes spawns faster and enemies tougher
- [ ] Gold costs / hero costs / economy are NOT affected by difficulty
- [ ] Difficulty can be changed mid-game from pause menu
- [ ] "Lock Difficulty" shows confirmation dialog
- [ ] After locking, difficulty cannot be changed
- [ ] Already-spawned enemies are NOT retroactively modified (only new spawns use the new multiplier)
- [ ] `python tools/qa_smoke.py --quick` PASS

### Feature 9: Dev Mode (Environment Variable)
**Owner:** Agent 05 (GameplaySystemsDesigner)
**Intelligence:** LOW

**What:** A developer/playtest mode activated by setting `$env:KINGDOM_DEV_MODE = "1"` in the console before launching the game. Makes gameplay testing new features fast by removing economic friction. This is NOT a player-facing feature — it's a dev tool.

**Design:**

**Activation:** Environment variable only, checked once at startup.
```python
# In config.py
DEV_MODE = os.getenv("KINGDOM_DEV_MODE", "0") == "1"
```

**Dev mode effects:**
- **Free heroes:** `HERO_HIRE_COST` overridden to 0 (hire as many as you want, instantly)
- **Infinite gold:** Starting gold set to 999,999
- **No guild cap:** `GUILD_MAX_HEROES` overridden to 999
- **Difficulty defaults to Easy** (can still be changed in-menu)
- **HUD indicator:** A visible "[DEV MODE]" label in the top-left or top-right corner of the screen so it's obvious when dev mode is active (prevents confusion during screenshot capture or recording)
- **Console log on startup:** `print("*** DEV MODE ACTIVE — free heroes, infinite gold ***")`

**Implementation:**
In `config.py` or early in `SimEngine.__init__()`, check `DEV_MODE` and override the relevant config values. Keep it simple — just override the numbers, don't create a parallel config path.

```python
# At the bottom of config.py, after all other config:
DEV_MODE = os.getenv("KINGDOM_DEV_MODE", "0") == "1"
if DEV_MODE:
    HERO_HIRE_COST = 0
    STARTING_GOLD = 999_999
    GUILD_MAX_HEROES = 999
    print("*** DEV MODE ACTIVE — free heroes, infinite gold ***")
```

**Usage (for Jaimie and agents):**
```powershell
$env:KINGDOM_DEV_MODE = "1"
python main.py
```

To disable, either unset the variable or set it to "0":
```powershell
$env:KINGDOM_DEV_MODE = "0"
python main.py
```

**Acceptance:**
- [ ] `$env:KINGDOM_DEV_MODE = "1"` activates dev mode
- [ ] Heroes cost 0 gold in dev mode
- [ ] Starting gold is 999,999 in dev mode
- [ ] No guild hero cap in dev mode
- [ ] "[DEV MODE]" visible on HUD when active
- [ ] Console prints dev mode banner on startup
- [ ] Without env var set, game behaves normally
- [ ] `python tools/qa_smoke.py --quick` PASS

## Phase 2: Playtest to v1.6.0 — Tuning Targets

After v1.5.9 ships, Jaimie playtests and we iterate on these knobs:

### Difficulty Curve Targets
- **Minutes 0-3:** Player hires first heroes, explores. Light lair trickle pressure. Tutorial feel.
- **Minutes 3-5:** First wave event. Player should feel "oh, I need to handle that." Survivable with 3-4 heroes.
- **Minutes 5-10:** Lair pressure builds. Multiple threats on screen. Player needs 6-8 heroes and bounties to direct them.
- **Minutes 10-15:** Mixed wave events. Player needs to build guardhouses near vulnerable buildings. Gold pressure forces choices.
- **Minutes 15-20:** Heavy waves. Player needs a full roster across multiple guilds. Strategic bounty placement critical.
- **Minutes 20+:** Boss waves. Only a well-managed kingdom survives. This is the "can you handle it?" zone.

### Tuning Knobs (All in config.py for easy iteration)
| Knob | Starting Value | Tune Range | Effect |
|------|---------------|-----------|--------|
| `HERO_HIRE_COST` | 100 | 75-150 | How scarce heroes feel |
| `STARTING_GOLD` | 1500 | 1200-2100 | Early game breathing room |
| `GUILD_MAX_HEROES` | 8 | 6-12 | Army ceiling |
| `MAX_ALIVE_ENEMIES` | 48 | 32-64 | Screen chaos level |
| `LAIR_INITIAL_COUNT` | 5 | 4-7 | Early pressure |
| `WAVE_EVENT_FIRST_MINUTE` | 3.0 | 2.0-5.0 | When danger starts |
| `WAVE_EVENT_INTERVAL_MINUTES` | 2.5 | 1.5-4.0 | Breathing room between events |
| `GUARDHOUSE_ARROW_DAMAGE` | 12 | 8-20 | Defense value |
| `GUARDHOUSE_ARROW_COOLDOWN` | 2.0 | 1.0-3.0 | Defense DPS |
| Lair spawn intervals | varies | ×0.5-×2.0 | Lair aggression |
| Difficulty multipliers (Easy) | HP 0.7, DMG 0.7, spawn ×1.5 | ±0.2 each | Easy mode feel |
| Difficulty multipliers (Hard) | HP 1.3, DMG 1.3, spawn ×0.7 | ±0.2 each | Hard mode feel |

**Note:** Difficulty tuning knobs are in `DifficultyConfig` in config.py. Phase 2 playtesting should validate that Normal feels right first, then verify Easy feels accessible and Hard feels punishing-but-fair.

### Playtest Protocol
1. Jaimie plays for 20+ minutes from fresh start
2. Notes: "When did I first feel threatened?" / "When did I feel overwhelmed?" / "When was I bored?"
3. Agent 01 translates notes into config adjustments
4. Repeat until the difficulty curve hits the targets above
5. When it feels fun: version bump to v1.6.0, changelog, release
